# Web UI

## Overview

The Web UI reads `.vscode/tasks.json` for each Git branch and provides a screen for executing tasks inside a run workspace prepared for that branch. Execution results are stored as run history, and you can later inspect the parent run state as well as per-child-task states and logs.

In the current implementation, the screen selection state is reflected in the URL. You can deep-link all the way down to branch, task, run, and selected child-task log.

## Assumptions and Layout

- The Web UI operates from the root of the target repository.
- The target repository is specified by `repository` in `runtask-ui.yaml`.
- The task list is generated by reading `.vscode/tasks.json` on the selected branch.
- Task execution resolves `.vscode/tasks.json` inside a run workspace prepared for the target branch.
- The configuration file is `runtask-ui.yaml`.

Typical paths in the repository are as follows.

- Config file: `runtask-ui.yaml`
- Task definitions to execute: `<repo>/.vscode/tasks.json`
- History storage root: `storage.historyDir/<runId>/`
- History index: `storage.historyDir/run-history-index.json` or an object with the same name in object storage

## `runtask-ui.yaml`

`runtask-ui.yaml` is the detailed specification intended for humans. If you use AI assistance or editor completion, use the following three resources together.

- `config-schema.json`: structure definition and completion for editors
- `llms.txt`: summary points for generative AI
- `.codex/skills/taskrun-ui-config/SKILL.md`: generation, editing, and review procedures for Codex

The source of truth is the Go validation in `internal/uiconfig/config.go`. The schema is for completion and structural validation, while `llms.txt` and the skill file are for AI assistance.

Generated or saved YAML is expected to start with the following schema comment.

```yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/shibukawa/vsc-taskrunner/main/config-schema.json
```

The main configuration sections are as follows.

### `server`

- `host`: host part of the HTTP server listen address
- `port`: listen port of the HTTP server
- `publicURL`: externally visible URL, also used when generating the OIDC callback URL

### `auth`

- `oidcIssuer`: OIDC issuer URL
- `oidcClientID`: OIDC client ID
- `oidcClientSecret`: OIDC client secret
- `noAuth`: disables authentication when set to `true`
- `sessionSecret`: signing key for the session cookie
- `allowUsers`: conditions that are allowed to access the UI, specified as `claim: glob` or `claim: [glob, ...]`
- `adminUsers`: conditions treated as administrators, specified as `claim: glob` or `claim: [glob, ...]`
- `apiTokens.enabled`: enables access tokens for batch execution
- `apiTokens.defaultTTLHours`: default TTL for newly issued tokens
- `apiTokens.maxPerUser`: maximum number of simultaneously valid tokens per user
- `apiTokens.store.backend`: token storage backend, either `local` or `object`
- `apiTokens.store.localPath`: storage file used when `backend: local`
- `apiTokens.store.object.*`: object storage settings dedicated to token storage when `backend: object`; these are separate from history storage settings in `storage.object.*`

Example:

```yaml
auth:
  allowUsers:
    email: "*@example.com"
    role:
      - runner
      - ops-*
  adminUsers:
    role:
      - administrator
  apiTokens:
    enabled: true
    defaultTTLHours: 720
    maxPerUser: 10
    store:
      backend: local
      localPath: .runtask/api-tokens.json
```

### `repository`

- `source`: target repository, either a remote URL or a local Git path
- `localPath`: local storage path where a clone is created when `source` is a remote URL
- `auth`: authentication settings used when reading a remote URL with a token

In the current implementation, the extracted path for tasks-sparse execution is fixed to `.vscode`, normal run workspaces always use a full checkout, and fetch depth is fixed at `1`. These cannot be changed from the config file.

When `source` is a remote URL, the current implementation runs `git clone --no-checkout` at UI startup and reuses an existing clone if present. During `/api/git/fetch`, the corresponding local branch is synchronized from `origin/*`.

If `repository.auth.type=envToken` is configured, token validation runs only when the token is present in the environment variable, both at startup and before `fetch`. If the environment variable is not set, no auth header is attached and access is attempted as a public repository. If a token is set, tokens that cannot be validated, or that are scoped too broadly, are rejected in fail-close mode.

The main fields are as follows.

- `type`: currently only `envToken` is supported
- `provider`: `github`, `gitlab`, or `bitbucket`
- `tokenEnv`: name of the environment variable that provides the token
- `baseURL`: base API URL; for GitLab Self-Managed this is usually `https://<host>/api/v4`
- `repo`: allowed repository identifier; `owner/repo` for GitHub and Bitbucket, `group/project` for GitLab
- `allowedHosts`: allowlist of permitted remote and API hosts
- `rejectBroadScope`: rejects tokens that can read repositories other than the configured target when set to `true`
- `requireReadOnly`: rejects tokens with write or admin-equivalent privileges when set to `true`

Notes:

- Currently only HTTPS token authentication is supported. SSH is out of scope.
- GitHub accepts only fine-grained PATs.
- GitLab requires `read_api` in addition to `read_repository` because validation is strict.
- Bitbucket assumes repository-scoped tokens.

If `allowUsers` is empty, any successfully authenticated user can access the UI. If `adminUsers` is empty, there are no administrator users.

When `apiTokens.enabled=true`, users matching `adminUsers` can issue access tokens from the user menu in the Web UI. Issued tokens can be used as `Authorization: Bearer <token>` for `GET /api/me`, `GET /api/runs`, `POST /api/runs`, and run detail APIs.

### `branches`

- Glob patterns for branch names that are allowed to be shown and executed

If `branches` is empty, only top-level branches are allowed.

### `tasks`

- Per-task UI settings keyed by a glob matching the task label
- `artifacts`: array of artifact rules; if not empty, artifact handling is considered enabled
- `artifacts[].path`: path or glob to collect
- `artifacts[].format`: either `zip` or `file`; default is `zip`
- `artifacts[].nameTemplate`: zip artifact name used when `format: zip`; default is `artifacts.zip`; ignored for `format: file`
- `preRunTask`: preprocessing steps that run once before the parent task
- `worktreeDisabled`: when `true`, the task runs in a tasks-sparse workspace that expands only `.vscode`

If `tasks` is empty, no tasks are allowed.

### `execution`

- `maxParallelRuns`: upper limit for concurrently running parent runs; `0` means unlimited, not disabled

### `storage`

- `historyDir`: root directory for run storage
- `historyKeepCount`: number of completed runs to keep per `branch + task`
- `backend`: storage backend for the history index, either `local` or `object`
- `object.*`: S3-compatible object storage settings used when `backend: object`

Defaults are defined in `internal/uiconfig/config.go`. Representative values are:

- `execution.maxParallelRuns`: `4`
- `storage.historyDir`: `.runtask/history`
- `storage.historyKeepCount`: `100`
- `storage.worktree.keepOnSuccess`: `0`
- `storage.worktree.keepOnFailure`: `3`

## Authentication and Authorization

When `auth.noAuth=false`, the Web UI requires OIDC login. Claims of the logged-in user are evaluated against `allowUsers`, and access is denied if they do not match.

If `allowUsers` is empty, any successfully authenticated user can access the UI. Only tasks defined in `tasks` are shown and executable.
- `POST /api/runs` is rejected with `403`

When `auth.noAuth=true`, authentication is disabled. In `/api/me`, the user is treated as `authenticated=false` and `canRun=true`.

## Screens and URLs

The current UI is roughly composed of the following areas.

- Branches: list of branches
- Tasks: list of tasks
- Inputs: execution-time inputs
- Runs: history list of parent runs
- Run Detail: summary of the selected run
- Dependency view: dependency graph reachable from the selected task
- child task log: log of the selected child task

Canonical URLs have the following shapes.

- `/branches/:branch`
- `/branches/:branch/runs/:runId`
- `/branches/:branch/tasks/:task`
- `/branches/:branch/tasks/:task/runs/:runNumber`
- `/branches/:branch/tasks/:task/runs/:runNumber/logs/:childTask`
- `/runs/:runId`
- `/runs/:runId/logs/:childTask`

The current implementation has the following properties.

- `branch`, `task`, and `childTask` are handled as full path segments and are URL-encoded as a whole.
- Switching the branch or task clears the currently displayed `runNumber` and selected child task.
- Run detail view can be separated from the selection state on the left. `/runs/:runId` shows a fully standalone run view, while `/branches/:branch/runs/:runId` keeps only the branch selection.
- URLs with `runNumber` stay pinned to that exact `branch + task + runNumber`. They do not automatically move to the latest run.
- If a task or child task specified in the URL does not exist in the current branch's `.vscode/tasks.json`, the UI shows a warning.
- However, if saved child-task state or logs exist in past runs, history display continues.

## Execution Flow

### Fetching the Task List

When a branch is selected, the frontend calls `GET /api/git/branches/{branch}/tasks` and reads `.vscode/tasks.json` on the target branch. If the branch has no `tasks.json`, the current implementation returns an empty array.

This API returns the following information for each task.

- `label`
- `type`
- `group`
- `dependsOn`
- `dependsOrder`
- `hidden`
- `background`
- `inputs`

`inputs` includes only inputs that are actually referenced by the task as `${input:...}`.

### Execution-Time Inputs

The current Web UI supports the following two input types.

- `promptString`
- `pickString`

Input values are sent as `inputValues` to `POST /api/runs` and used during task resolution for `${input:...}`.

### Execution

Task execution sends the following JSON to `POST /api/runs`.

```json
{
  "branch": "feature/demo",
  "taskLabel": "npm-build",
  "inputValues": {
    "target": "web"
  }
}
```

On the server side, processing happens in the following order.

1. Atomically update the history index to generate `runId`, assign `runNumber` per `branch + task`, and register the run as `running`
2. Prepare the backing storage path as `history/<runId>/` in local mode or `runs/<runId>/` in object storage mode
3. Prepare the run workspace for the target branch
4. Read `.vscode/tasks.json` inside the run workspace, apply `inputValues`, and resolve tasks
5. Execute `tasks.<pattern>.preRunTask` for the selected task in order
6. Execute the selected task
7. Collect artifacts for each artifact rule after execution
8. Update `meta.yaml` and the history index to a completed state, then delete run payloads for `runId`s removed because of retention limits

Dependent tasks are processed according to the existing runner's `dependsOn` and `dependsOrder` behavior.

- If `dependsOrder` is omitted, execution is `parallel`
- If `dependsOrder: sequence` is specified, execution is sequential

Dependent tasks that were not executed because of a dependency failure are stored as `skipped`.

`preRunTask` is not stored as a child task. It appears only in the combined log, and if it fails, the main task is not started and the whole run is marked failed.

Artifact collection processes each rule independently in definition order. With `format: zip`, one zip is created per rule. With `format: file`, each matched file is stored individually. If the same file matches multiple rules, it is included in each corresponding output.

## History and Stored Data

The storage path for each run is `storage.historyDir/<runId>/` in local mode and `<prefix>/runs/<runId>/` in object storage mode. Each run directory contains the following.

- `meta.yaml`
- `stdout.log`
- `tasks/*.log`
- `worktree/`
- `artifacts/`

### History Index

`GET /api/runs` on the top page does not scan all run directories. It reads only a single history index.

The index stores the following information per `branch + task`.

- `nextRunNumber`
- list of latest runs
- summary information for each run list entry: `runId`, `runKey`, `branch`, `taskLabel`, `runNumber`, `status`, `startTime`, `endTime`, `exitCode`, `user`, `hasArtifacts`

Actual artifact paths and detailed child-task data are not stored in the index. Details are read from each run's `meta.yaml` and artifact directory.

In local mode, index updates take a file lock and serialize concurrent execution within a single host. In object storage mode, updates use version or ETag compare-and-swap.

### `meta.yaml`

`meta.yaml` contains metadata for the parent run and child tasks. Main fields are:

- `runKey`
- `runId`
- `branch`
- `taskLabel`
- `runNumber`
- `status`
- `startTime`
- `endTime`
- `exitCode`
- `worktreeKept`
- `artifacts`
- `user`
- `inputValues`
- `tasks`

`tasks` contains per-child-task state. Main fields are:

- `label`
- `dependsOn`
- `dependsOrder`
- `status`
- `startTime`
- `endTime`
- `exitCode`
- `logPath`
- `historical`

### Log Files

- `stdout.log`: combined log for the parent run
- `tasks/*.log`: log per child task

Child task log file names are not created directly from task labels. They are created using sanitized names for storage.

## Worktrees and Retention Policy

Each run uses its own dedicated workspace. The workspace payload is stored at `historyDir/<runId>/worktree/` in local mode and `runs/<runId>/worktree/` in object storage mode.

By default, a full run workspace is prepared. If the selected task has `worktreeDisabled: true`, a tasks-sparse workspace is prepared instead, expanding only `.vscode` rather than the full run workspace. This is intended for jobs that can run using only `.vscode/tasks.json` and a minimal set of helper files.

In a tasks-sparse workspace, tasks or pre-run tasks that depend on files outside `.vscode` may fail.

After a run completes, worktree retention is managed according to `storage.worktree.keepOnSuccess` and `storage.worktree.keepOnFailure`. Old worktrees outside the retention window are deleted during prune, and `worktreeKept` in `meta.yaml` is updated as well.

`storage.historyKeepCount` is the number of completed runs kept per `branch + task`. Old run directories are deleted according to this limit.

## Artifacts

Artifacts are files collected from a run's worktree according to per-task `artifacts` rules. If `artifacts` is not empty, artifact handling is considered enabled. Each rule can have `path`, `format`, and `nameTemplate`.

With `zip`, directory targets are expanded recursively.

- `dist`: recursively stores contents while keeping `dist/...` inside the zip
- `dist/*`: expands direct children of `dist` into the root of the zip, while recursively storing their descendants

With `file`, directory targets are not allowed. Use explicit file paths such as `dist/index.html`. Even when a wildcard is used, the run fails if the result includes a directory.

- `zip`: stores files matched by the rule in a single zip archive. The default name is `artifacts.zip`.
- `file`: stores each file matched by the rule individually while preserving its relative path.

For `zip`, the archive name can be customized per rule with `nameTemplate`. The following six placeholders are available.

- `{yyyymmdd}`: UTC date of run start
- `{hhmmss}`: UTC time of run start
- `{yyyymmddhhmmss}`: UTC datetime of run start
- `{hash}`: first 7 characters of the worktree HEAD commit hash
- `{longhash}`: full worktree HEAD commit hash
- `{branch}`: branch name normalized into a filename-safe string

Example: `frontend-{branch}-{yyyymmdd}-{hhmmss}-{hash}.zip`

If the same file matches multiple rules, it is included in every corresponding output. `nameTemplate` is ignored for `format: file`.

The artifact copy destination is as follows under each run.

- local: `storage.historyDir/<runId>/artifacts/`
- object storage: `<prefix>/runs/<runId>/artifacts/`

For `zip`, `artifacts/artifacts.zip` is stored. For `file`, files are stored under `artifacts/` while preserving their relative paths. In both cases, they are recorded in `meta.yaml` under `artifacts`.

## Logs and Realtime View

The browser subscribes to SSE from `GET /api/runs/{runId}/log`. The page URL itself keeps `runNumber`, but the UI resolves `runId` from the list API and then calls detail, SSE, and worktree APIs using that resolved ID.

Main events are as follows.

- `task-start`
- `task-line`
- `task-finish`
- `task-skip`
- `done`

On the UI side, child-task logs are filtered and displayed using `taskLabel` in the event payload. After a run completes, saved `tasks/*.log` files can still be used for redisplay.

## Supporting APIs and Browsing

The main APIs used for browsing are as follows.

- `GET /api/runs`
  - Returns the parent run list from the history index.
- `GET /api/runs/{runId}`
  - Returns details for one run, including child-task state.
- `GET /api/runs/{runId}/worktree`
  - Returns the list of saved worktree files.
- `GET /api/runs/{runId}/worktree/{path...}`
  - Returns file contents inside the saved worktree.

If the worktree has already been pruned, the worktree browsing APIs cannot be used.

## Verifying with Compose

Local history mode:

- `docker compose up -f docker-compose.yml`

Object storage mode using RustFS:

- `docker compose -f compose.object.yaml up`

In object storage mode, `runtask-ui.object.yaml` is used, and the history index and `runs/<runId>/...` are written to the RustFS S3 endpoint. Local staging is used while the run is executing, and data is resynchronized to object storage on completion and after worktree prune.

The current `compose.object.yaml` is a local verification setup using a single drive layout. It has no availability, redundancy, or disk failure tolerance. Use it only for lightweight S3-compatible API verification.

## Notes

This document describes actual operational behavior based on the current implementation. It is not an ideal future design. The description is grounded in what `internal/web` and `internal/uiconfig` actually do today.
